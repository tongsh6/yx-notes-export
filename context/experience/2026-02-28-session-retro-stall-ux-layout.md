# 会话复盘：导出“卡住感”与 GUI 布局打磨

## 背景
本次会话同时处理了两类问题：
- 导出在单条笔记阶段长时间无进度，用户感知为“卡死”
- GUI 出现按钮挤压、文字不全、对齐不一致

## 本次会话暴露的问题
- 早期 UI 只在“切下一条笔记”时更新，单条内部耗时没有活动反馈
- 顶部工具按钮分散在不同区域（主题在全局头，清空在日志局部头），视觉上不统一
- 多处固定宽度叠加中文文案，导致窄窗口和高 DPI 下更易截断
- 一次 `task()` 调用遗漏 `run_in_background` 参数，产生无效调用
- 测试桩与真实线程对象差异，暴露出信号连接兼容性边界

## 可沉淀经验
- “看起来卡住”不等于死锁，先区分无反馈与不可推进
- 长任务要有三层交互：活动状态、无进度提示、可操作（停止/跳过）
- 协作式取消优先于强杀线程，避免 GIL/资源状态不一致
- 中文 GUI 不要过度依赖固定宽度，优先最小宽+弹性布局+统一高度
- 结构化日志需要覆盖 `api_wait_retry` / `export.stall_hint`，方便排障复盘

## 本次会话的改进点
- 在需求进入“视觉打磨”阶段时，先给出界面验收清单（对齐/截断/节奏/层级）
- GUI 变更增加截图回归（固定窗口尺寸 + 关键区域对比）
- 对长任务交互先画状态机，再实现（idle/running/stall/stopping/skipping/done）

## Best Practice（可复用）
- 先可见，再可控：先让用户知道系统在做什么，再提供停止/跳过
- 状态优先于百分比：长任务中状态文案比进度条更有诊断价值
- 一次改一层：稳定性修复与视觉优化分批提交，降低回归风险
- UI 改动后必须跑全量回归：LSP + pytest + 关键手测场景

## 影响
- 导出过程从“黑盒等待”变为“可观察 + 可干预”
- GUI 在中等窗口宽度下的对齐与可读性明显提升
- 后续同类需求可直接复用 skill/template/checklist/scripts 资产
